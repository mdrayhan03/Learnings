A: if a object constructs with many fields which are optional it becomes a problematic, because some parameters we don't need but for construction we need to pass them like None and we have to remember the parameters serial is serial mis our code can break or can be logical error and Builder Pattern solves this problem. In Builder Pattern we use method chaining to construct object so we don't need the serial or parameters and we can only call which parameters do we need and no need to call optional unnecessary parameters method.
B: In Builder Pattern first we make all the object fields with None or necessary default parameters for optional fields, then in we build parameter by parameter like lego pieces joining one by one and at the last becomes the complete picture same for this pattern we build one by one parameters method if need else we use that default value parameter and at the last we use build function to make the entire object and return.

A: in a multithreaded environment if we use singleton it's not use as singleton because every thread use their own object we can think it like that one thread use one object for it's entire work but if 5 thread we create 5 object, to solve this we have to use thread locking so that at a same time two thread cannot create object and if object is created always use the created object don't create new and for multiprocess environment singleton is quite impossible.
B: in a game engine a monster object can be heavy loading object like loading animation and assets can take 5-10s, every time new monster instantiation take the same 5-10s if we use prototype pattern using deepcopy, it's not like normal copy, in a normal copy it's only copy the first layer and if the main object change the copy object also change, in deepcopy it's copy the entire object from first to end all layer so it's becomes seperated then main object and work as entirely individual and from prototyping we don't need 5-10s time (it depends on scenario) because all static heavy loaded work done in main object and we use those in the prototyped object.
